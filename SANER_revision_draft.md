## 0. Data-quality audit — three reproducibility bugs found and fixed

While reconstructing the dataset to add RQ4, I found and fixed three
bugs in the aggregation pipeline that affected the paper's headline numbers.
Flagging this prominently rather than burying it, since the fixes are
themselves a legitimate rigor contribution for the "Verifiability" section
reviewers flagged (B and C both raised replication-package concerns).

**Bug 1 — invalid repo in the original 100.** `data` (owner: `owid`) does not
meet the paper's own selection criteria (>10,000 stars, >9,000 forks, ≥10
years old): no repository owned by `owid` exceeds 5,664 stars, and the
locally-extracted data for this entry shows only 234 commits over the full
window. This looks like a data-entry error from the original repo-selection
step, not a renamed/moved repository. Replaced with `microsoft/vscode`
(187k stars, 41k forks, created 2015, active, GitHub-native workflow) and
ran the full original extraction pipeline against it.

**Bug 2 — non-deterministic file selection in `combine_doc_commit_ownership.py`.**
Every per-repo output folder contains two ownership-summary files: a
canonical `{repo}_2020_2025_health_docs_ownership_summary.csv` and a legacy
`5yr_health_docs_ownership_summary.csv` left over from an earlier pipeline
iteration. The combiner picked whichever file its filesystem happened to
list first — unordered, not guaranteed stable across machines or runs — and
the two files disagree numerically (e.g. `go`: 36 vs. 34 contributors,
Bus-50 of 13 vs. 12) despite having identical schemas. This affected the
*selection*, not the values themselves, for 24 of 100 repos. Fixed by
anchoring the combiner on the canonical filename deterministically, then
regenerated ownership data fresh (current script version) for all 100 repos
to remove any doubt about provenance. One repo (`FFmpeg`) additionally had a
canonical file with a **stale schema** (missing a column, i.e. generated by
an older script version) — also regenerated.

**Bug 3 — "longest filename wins" heuristic in `combine_doc_entropy_extraction.py`.**
This combiner preferred the *longest* matching filename as a "more specific
= better" heuristic. For `AdminLTE` specifically, a stale duplicate run left
a typo'd file (`AminLTE_2020_2025_health_docs_entropy_summary.csv`, note the
missing "d") whose longer name caused it to be picked over the correct
`AdminLTE_2020_2025_entropy_summary.csv` — 14 vs. 30 documentation commits,
entropy 0.407 vs. 0.615. This is what first surfaced the bug: `AdminLTE`'s
rhythm-script commit count (30, from `Doc_rhythm.py`) didn't match its
entropy-script commit count (14, from the mis-selected file) despite both
scripts sharing byte-identical health-file matching rules. Fixed the same
way as Bug 2; confirmed isolated to this one repo across all 100.

**Net effect on headline numbers:**
- Archetype distribution: **Consistent 32%, Occasional 48%, Sparse 20%**
  (was 32/51/17). 6 of 100 repos changed label: `AdminLTE` moved
  Sparse→Occasional (the direct Bug-3 fix); `erpnext`, `odoo`, `reveal.js`,
  `youtube-dl`, `zxing` moved Occasional→Sparse as a second-order effect of
  the corrected data shifting cluster boundaries (their own values didn't
  change — the centroids did).
- Silhouette score at k=3: **0.590** (was 0.603) on the non-bot-filtered

- RQ2/RQ3 confounding-control conclusions (Sections 2-3 below) are

  F=0.91/p=0.41); rhythm metrics still don't add power to bus factor

> **4.2.3 Robustness to Artifact-Type Heterogeneity.** Table 1's 22 health
> documentation artifacts span heterogeneous purposes, from actively-maintained
> process documentation (README, CHANGELOG) to attribution and legal artifacts

> COPYING, AUTHORS, CREDITS, THANKS, CONTRIBUTORS) — and recompute entropy and
> AWR within each stratum independently.
>
> At the ecosystem level, 88.6% of the 14,919 documentation-touching commits
> touch only living/process files, 9.4% touch only static/attribution files,
> and 2.0% touch both; the median repository's static-doc share of
> documentation activity is 7.1% (mean 12.9%, IQR 0.9%–18.2%). Combined-type

> indicating the reported metrics are overwhelmingly driven by living/process
> documentation rather than by the inclusion of rarely-touched static files.

> living-only entropy and AWR (same k-means procedure as Section 4.1.3) and

> touched 157 times) rather than manual governance activity. Because
> `Doc_rhythm.py`/`doc_entropy.py` (unlike our ownership pipeline) do not
> filter bot commits, these repositories' archetype labels are partly
> influenced by automated attribution-file churn.
>

> identity for all 100 repositories and re-ran the rhythm computation
> excluding bot-authored commits (same bot-detection patterns as the
> ownership pipeline). At the ecosystem level, 1,082 of 14,949
> documentation-touching commits (7.2%) are bot-authored. Re-clustering on
> bot-filtered entropy and AWR changes the archetype label for 5 of 100
> repositories (95% raw agreement, Adjusted Rand Index = 0.855 — higher

### 1a. Finer-grained artifact taxonomy: what kind of documentation actually gets touched

*Extends the living/static binary split above into four categories,
directly addressing reviewer A's concern that Table 1's 22 artifact types
"serve very different purposes" and may be misrepresented by an aggregate
statistic. Suitable as a new descriptive subsection (e.g., 4.2.4) — this
result is solid and low-risk, unlike the exploratory items elsewhere in
this document.*

We partition the 22 health documentation artifacts into four purpose-built
categories: **onboarding/process** (README, CONTRIBUTING, BUILDING,
COMMIT_CONVENTIONS, PULL_REQUEST_TEMPLATE, ISSUE_TEMPLATE), **governance/
policy** (GOVERNANCE, CODE_OF_CONDUCT, SECURITY, SUPPORT, MAINTAINERS,
ROADMAP, VISION), **change-tracking** (CHANGELOG, HISTORY, RELEASE(S)), and
**legal/attribution** (LICENSE, NOTICE, COPYING, AUTHORS, CREDITS, THANKS,
CONTRIBUTORS).

**The ecosystem-level and per-repository pictures diverge sharply, and the
divergence itself is worth reporting.** At the ecosystem level, change-
tracking accounts for 49.4% of all 14,949 documentation-touching commits —
appearing to be the dominant category. But the **median per-repository**
share of change-tracking commits is **0%**, and 60 of 100 repositories have
*zero* change-tracking activity across the entire 5-year window. The
ecosystem-level statistic is driven by a small number of repositories with
heavy, often automated, changelog activity (e.g., high-frequency release-
note bots), not by change-tracking being a typical or widespread practice.
The category that *is* typical: **onboarding/process** documentation
accounts for a median 73.3% of a repository's documentation commits and is
present in all 100 repositories (zero missing) — this is what "documentation
maintenance" actually consists of for the median project. Governance/policy
(median 3.4% share, absent in 30 repos) and legal/attribution (median 6.7%,
absent in 18 repos) are minor and frequently untouched entirely.

| Category | Ecosystem share | Median per-repo share | Repos with zero activity |
|---|---|---|---|
| Change-tracking | 49.4% | 0% | 60/100 |
| Onboarding/process | 40.1% | 73.3% | 0/100 |
| Legal/attribution | 11.4% | 6.7% | 18/100 |
| Governance/policy | 5.4% | 3.4% | 30/100 |

**Recommended framing**: this is direct evidence for a point reviewer A
raised generally (artifact-type aggregation can misrepresent what's
actually being measured) and gives it a precise, quantified answer:
aggregate ecosystem-level statistics in this paper (e.g., the 68.4%
DocOnly rate, the archetype distribution) are numerically dominated by
onboarding/process documentation specifically, not by "documentation" as an
undifferentiated whole, and the paper should say so explicitly rather than
let the aggregate stand unqualified.

**We additionally tested each category's entropy, AWR, and participation
rate against the two validated RQ4 findings (staleness, contributor loss)
separately, to check whether slicing by category surfaces a stronger or
different signal than the all-docs aggregate.** It does not: no category
produces a result that survives the full robustness battery (one candidate
— legal/attribution entropy vs. legal/attribution staleness — passed all
four checks initially, p=0.002-0.029, but dissolved on inspection: the
median repository has only 4 legal/attribution commits total over 5 years,
and restricting to repositories with ≥10 such commits took the effect from
p=0.03 to p=0.64. Sparse per-category commit counts make entropy
calculations unstable at this granularity — not a real effect). The
category breakdown is valuable as description, not as a source of
additional predictive-validity claims; the two validated RQ4 findings
should stay reported at the all-documentation-types level where they were
established.

## 3. RQ3 reframing: bus factor vs. rhythm, controlling for size

*Replaces the sentence in 6.2.3 beginning "However, Bus-50 also correlates
independently with entropy... indicating that consistent documentation rhythm
predicts more distributed ownership even after accounting for project size."*

**What changes:** the original claim treats bivariate Spearman correlations
(ρ=+0.30 with entropy, ρ=+0.29 with AWR) as evidence of an effect
*independent* of project size — but a Spearman ρ is not a partial
correlation, so this oversells what was actually tested. A proper multivariate
test (OLS of log(Bus-50) on log(contributor count) + entropy + AWR, and each
individually to avoid the entropy/AWR collinearity — r=0.90 between them)
shows neither rhythm metric adds explanatory power once contributor count is
in the model (jointly: F=0.136, p=0.87; individually: entropy p=0.87, AWR
p=0.70). Contributor count alone explains R²=0.279 of log(Bus-50) variance;
adding both rhythm metrics moves this to R²=0.281.

> Suggested replacement text: "Bus-50 correlates strongly with total
> contributor count (ρ=+0.62, p<0.0001). Entropy and active window rate each
> show a positive bivariate correlation with Bus-50 (ρ=+0.30, ρ=+0.29,
> p<0.005) that we initially interpreted as evidence rhythm predicts ownership
> distribution independent of project size. A multivariate test does not
> support that interpretation: regressing log(Bus-50) on log(contributor
> count) together with entropy and AWR shows neither rhythm metric adds
> significant explanatory power beyond contributor count alone (individually
> or jointly; all p>0.6). We therefore withdraw the independence claim:
> contributor count is the dominant and, on this evidence, sufficient
> predictor of documentation bus factor among the metrics tested here. This
> does not undermine the core structural-fragility finding (31% of
> repositories have Bus-50=1 regardless of maturity/size selection criteria)
> but clarifies that *documentation-specific* rhythm is not, by itself,
> a distinguishing driver of ownership concentration beyond what community
> size already predicts."

**What still holds, unchanged:** the paper's core RQ3 claim — documentation
ownership is *more concentrated than code ownership at the same repository* —
survives this check. The excess concentration (doc top-3 share minus code
top-3 share, computed within each repository, which already nets out most
size effects) is not significantly correlated with contributor count
(ρ=0.157, p=0.118), and 73/100 repositories still show positive excess
concentration. This within-repository comparison is a stronger design than
the cross-repository bus-factor-vs-rhythm comparison above precisely because
each repository serves as its own size-matched control — worth stating
explicitly in the methods to preempt the same critique being raised again.

## 4. New RQ4 — Does documentation governance predict downstream outcomes?

**Add as a new Section 7** (RQ1-3 stay as Sections 4-6; renumber Discussion/
Threats accordingly). This is the paper's strongest new contribution and
should be positioned as such in the abstract and intro, not buried as a
robustness appendix.

> **RQ4: Does documentation governance predict OSS project outcomes, beyond
> what contributor ecosystem size alone predicts?**

**Motivation.** RQ1-3 characterize documentation governance descriptively.
Reviewers across both submissions (implicitly ASE, explicitly the framing
critique that prompted this section) asked the natural next question:
*so what?* — do these metrics predict anything, or are they measurement for
measurement's sake? We answer this directly, testing rhythm (entropy,
active window rate), ownership (documentation bus factor), and a fourth
governance dimension — **documentation participation rate**, the share of a
project's overall contributor base that ever touches health documentation —
against a battery of outcomes spanning social/coordination signals *and*
documentation health itself.

**Method.** For each of four continuous governance predictors (entropy,
AWR, log-transformed Bus-50, participation rate) and each outcome, we fit
`outcome ~ log(contributor_count) + predictor` and report whether the
predictor adds explanatory power over the size-only baseline (nested-model
F-test), consistent with the confound-control approach in Sections 5-6.
Outcomes tested:

1. Newcomer retention (returns in a 2nd distinct month) — RQ4a
2. Maintainer/veteran turnover (established contributors going inactive in
   the window's final year)
3. Contributor-count growth (year 1 → year 5 of the window)
4. Repository activity trend (slope of monthly commit volume)
5. Issue close time and PR merge time (GitHub API)
6. **Documentation staleness** — median days-since-last-touch across a
   repository's health documentation files, as of the window end

Every candidate finding was additionally stress-tested — robust (HC3)
standard errors, Cook's-distance outlier removal, and rank-based partial
correlation — before being reported, after an initial candidate (excess
ownership concentration vs. entropy) looked significant under plain OLS
(p=0.023) and evaporated entirely once 3 influential points were removed
(p=0.396). We report only what survives this stress test.

**Result: five of six outcomes are null or fragile.** Retention (F=0.16,
p=0.958), turnover (p=0.87-0.90), growth and activity trend (significant
under plain OLS, p<0.002, but collapse under robust SE and outlier removal
to p>0.17), and issue/PR resolution latency (p=0.41-0.72) show no
independent effect of any governance predictor once contributor count is
controlled.

**Result: documentation participation rate independently predicts
documentation staleness — and it survives everything.** Repositories in the
Consistent archetype keep documentation roughly twice as fresh as the other
two archetypes (median 325.5 vs. 689 / 772.5 days since last touch;
Kruskal-Wallis p=0.027; Consistent vs. Sparse p=0.015, Consistent vs.
Occasional p=0.040). At the continuous-predictor level, participation rate
predicts lower staleness controlling for contributor count (p=0.002), holds
under robust SE (p=0.031) and after removing the 3 most influential repos
(p=0.042) — and, critically, **survives controlling for raw documentation
commit volume as well** (p=0.024), which entropy and AWR do not (p=0.74 and
p=0.50 once volume is added as a control). This last check matters: entropy
and AWR's raw relationship with staleness is largely explained by "repos
that produce more documentation commits naturally spread across more months
and more files" — an unsurprising volume artifact. Participation rate's
relationship is not — it holds even when comparing repositories that
produce the *same amount* of documentation work. The distinguishing factor
is not how much documentation effort a project puts in, but **how broadly
that effort is distributed across contributors**: a repository where many
different people touch documentation keeps it fresher than one where the
same volume of work is concentrated in one dedicated documenter, at any
fixed level of overall effort. In a joint model with entropy, AWR, and
participation rate together, the governance predictors jointly explain
15.0% of staleness variance vs. 0.6% for contributor count alone (F=5.38,
**p=0.0018**).

**Framing for the paper**: this is the concrete, validated "so what" the
paper needs. State it plainly: *"We show that documentation participation
rate — the breadth of contributor involvement in documentation, not its
raw volume or temporal rhythm — independently predicts whether a project's
documentation stays current, beyond both community size and effort volume.
This is, to our knowledge, the first validated predictor of documentation
freshness derived from governance structure rather than content analysis,
and it directly motivates tracking participation rate (not just bus factor)
as a CHAOSS-style health signal: projects can have healthy rhythm and low
concentration risk while still accumulating stale documentation if too few
contributors are involved in maintaining it."* This is a specific,
mechanism-grounded, actionable claim — exactly the "I could build this into
CHAOSS tomorrow" reaction a strong empirical SE paper aims for, and it
earned that status by surviving a genuinely adversarial validation process
rather than being asserted from a single significant p-value.

**What NOT to claim**: entropy and AWR, despite being the paper's headline
RQ1 metrics, are **not** independently validated predictors of anything
tested here once volume/size are controlled — be precise that the validated
claim is about *participation rate* specifically, not "rhythm archetypes"
broadly. Overextending this finding to entropy/AWR would not survive the
same scrutiny that got the original submission rejected the first time.

### 4b. Second validated result: documentation bus factor predicts documentation-newcomer retention

A construct-matched retention test — predicting whether newcomers *to
documentation specifically* return to touch documentation again, rather
than the domain-mismatched "any commit at all" outcome used elsewhere in
this section — found that **documentation bus factor (Bus-50) predicts
documentation-newcomer retention**, controlling for contributor count,
total documentation-commit volume, and the raw number of newcomers itself
(the strongest confound candidate, since Bus-50 and newcomer count
correlate at ρ=0.63). The relationship survives robust SE, outlier removal,
and gets *stronger*, not weaker, when restricted to repositories with at
least 10 documentation newcomers (n=75; p=0.0002, robust HC3 p<0.0001,
p<0.0001 after outlier removal).

By bus-factor tercile: repositories with concentrated documentation
ownership (low Bus-50) show a median newcomer retention rate of 33.3%;
moderately distributed ownership, 26.4%; highly distributed ownership,
13.6% (Kruskal-Wallis p=0.00005).

The direction is worth stating precisely, since it complicates rather than
simply reinforces the paper's RQ3 framing: **lower** bus factor — the
condition RQ3 identifies as structurally risky (single point of failure) —
is associated with **higher** newcomer retention. Concentrated ownership
and newcomer retention move together, not apart. A plausible mechanism:
a single identifiable maintainer is easier for a newcomer to get a response
from than a diffuse group with no clear point of contact, but this analysis
cannot distinguish that from reverse causality (e.g., smaller, more
intimate communities may independently produce both concentrated ownership
and higher retention through a shared underlying cause not fully captured
by the community-size control). Report as an association, not a causal
claim, and pair it explicitly with the RQ3 fragility framing as a tension:
the same structural feature that reduces resilience to losing a maintainer
may be part of why that maintainer's presence retains newcomers in the
first place. This does not change the paper's fragility argument (Bus-50=1
in 31% of repos is still a real risk) — it adds a second dimension to what
that risk trades off against.

**Convergent, properly-lagged replication in a second population.** The
above is contemporaneous (Bus-50 and retention measured over the same
5-year window). We additionally tested whether bus factor computed on only
the *first half* of the window predicts *documentation contributor loss*
(not gain) in the *second half* — a properly time-ordered design where the
predictor is fixed before the outcome period begins, and a second
population: established documentation contributors (first doc-touch in the
window's first half) rather than newcomers. Two outcome definitions —
loss rate among all established contributors, and among the top-quartile
("veteran") established contributors by volume — both show the same
relationship, both surviving the full robustness battery (size control,
volume control, robust HC3 SE, outlier removal): higher first-half bus
factor (more distributed ownership) predicts **more** contributor loss in
the second half (all established: p=0.0001-0.0006; top-quartile veterans:
p=0.0006-0.0009; ρ≈0.38-0.39 both). This is the same relationship as above,
not a separate claim, but it substantially strengthens it: the earlier
contemporaneous version cannot rule out reverse causality, while this
lagged version structurally cannot be explained backwards (the predictor
is fixed before the outcome period exists), and it generalizes beyond
newcomers to established contributors as well. Report the two together as
one finding with two lines of complementary evidence, not as separate
results.

*Methodological note worth keeping in the replication package*: an initial
version of this lagged test also included entropy and active window rate
as predictors and found what looked like strong effects (p<0.0001 for
several specifications). These turned out to be **circular**: full-window
entropy/AWR are computed from the same aggregate monthly commit vector that
the "did contributors depart" outcome partially determines (if established
contributors leave and aren't replaced, overall monthly activity drops,
which mechanically lowers full-window AWR/entropy) — predictor and outcome
were not independent. Recomputing entropy/AWR on the first half only (the
same fix applied to bus factor above) made the effect vanish entirely for
the all-established-contributor outcome (p=0.78-0.99) and reduced it to a
non-surviving, volume-control-failing pattern for the veteran-only outcome
(p=0.02 size-control, p=0.10-0.13 volume-control — not reported). Bus
factor was checked the same way and is not subject to the same mechanical
relationship in the same way entropy/AWR are, which is part of why it
survives, and part of why entropy/AWR should not be over-claimed even
though they occasionally look significant in a same-window specification.

### 4c. Exploratory result (weaker evidence): process-doc participation and issue response latency

A third pattern surfaced testing process-documentation-only (README,
CONTRIBUTING, CHANGELOG, GOVERNANCE, SECURITY, etc. — excluding static/
attribution files) participation rate against issue time-to-first-response,
pulled via the GitHub GraphQL API (first-comment timestamp minus issue
creation timestamp, for the 100 most recently closed issues per repository).
Controlling for contributor count, total process-documentation commit
volume, and total issue volume simultaneously: higher process-doc
participation rate is associated with **slower** first response to issues
(coef=2.44, p=0.033; robust HC3 SE p=0.028).

**This is presented as exploratory, not validated, evidence** — it is
markedly weaker than the two results above and does not clear the same bar.
Specifically: after removing the 3 most influential repositories (Cook's
distance), significance drops to p=0.078, and the partial rank correlation
controlling for all three confounds is weak (ρ=0.15). Unlike the two
findings above, this one does not survive the full robustness battery.

Inspecting the influential points directly rather than mechanically
discarding them: they are a genuine mix, not a one-sided artifact.
`spark` (participation=0.011, response=18 hours) fits the direction
cleanly. `todomvc` and `markdown-here` are very small repositories (8 and 2
total contributors) where participation rate is mechanically inflated by a
tiny denominator, both slow to respond — consistent with the direction but
for a reason that should raise caution (the same class of small-project
artifact that produced a false positive in the repository-inactivity
analysis, discarded above). `grafana`, however, is a genuine
counter-example: low participation (0.077) *and* very slow response (578
hours) — a large, high-issue-volume project that does not fit the pattern
at all. That the influential set contains both supporting and contradicting
cases, and that significance nonetheless depends on their inclusion, is
itself informative: the relationship may be real but is not strong enough
to be visible independent of a handful of extreme repositories in a
100-repo sample.

**Recommended framing if included**: report this explicitly as a secondary,
unconfirmed observation — e.g., in the discussion section as a direction
for future work with a larger sample, not alongside the two validated
findings in the results. State the numbers plainly, including the failed
outlier-robustness check, rather than presenting it with the same
confidence as Sections 4a/4b. If a reviewer would reasonably ask "does this
hold up to scrutiny?", the honest answer here is "partially" — say so.

## 5a. Bus factor robustness: file-centric ownership (Jabrayilzade-inspired)

*Addresses reviewer C directly: "Bus factor methodology follows the older
baseline by Cosentino et al. rather than the newer, more well-rounded
method by Jabrayilzade et al."*

A full Degree-of-Authorship model (Fritz et al. 2010, as adapted by
Jabrayilzade et al. 2022) needs line-level blame data we don't have. As a
tractable but genuinely different robustness check, we computed a
**file-centric** bus factor: for each health-doc file, identify its
top-commit-count owner (not aggregate commits across all files — per-file
primary ownership, which is Jabrayilzade et al.'s core methodological
departure from Cosentino et al.'s aggregate-commit approach), then compute
Bus-50/80 as the minimum number of distinct "primary file owners" needed to
collectively be the top owner of ≥50%/80% of touched health files.

**Result**: moderate positive correlation with the existing commit-count
Bus-50/80 (Spearman ρ=0.468, p<0.0001 for Bus-50; ρ=0.548, p<0.0001 for
Bus-80 — different enough to be a real robustness check, not a
reproduction). Critically, the file-centric method finds **more** structural
fragility, not less: 55.1% of repos (54/98) have a file-centric Bus-50 of 1,
compared to 27.6% under the commit-count method on the same 98 repos. This
suggests the paper's "31% of repositories depend on a single contributor"
claim is, if anything, a **conservative** estimate — concentrating on who
actually *owns* individual files (rather than who has the highest raw
commit count across all files) reveals single-point-of-failure risk in over
half of repositories. Recommend adding this as a robustness-check paragraph
in 6.2.3 — it directly strengthens the paper's core fragility claim while
answering reviewer C's specific methodological critique.

## 6a. External-linkage heuristic: reproducibility confirmed, but precision is a real concern

**Reproducibility — good news.** No raw commit message was persisted
anywhere in the original replication package (only a handful of truncated
examples), so the 49.0% external-linkage rate had never actually been
independently re-derivable. Using the full commit-history extraction, we
pulled the full message (`%B`, subject + body — matching
`commit_message_external_links.py`'s own `get_commit_message` exactly) for
all 14,919 documentation-touching commits and reclassified them with the
byte-identical heuristic. Result: **49.09% reactive**, reproducing the
paper's 49.0% almost exactly, and the sub-pattern breakdown matches closely
(issue-number references 83.3% of reactive commits — an exact match to the
paper's reported 83.3%; GitHub PR URLs 18.3% vs. reported 17.7%; fix-with-
error-term 6.0% vs. reported 5.7%). **The ecosystem-level statistic is
reproducible and was computed correctly.**

**Precision — a real concern, not previously checkable.** Reviewer B
specifically asked for validation of this heuristic (annotator procedures,
agreement measures, precision/recall) and it was never possible to check
before, for the reason above. We manually coded a stratified random sample
of 40 heuristic-classified "reactive" commits (from the 75 pulled) against
whether the commit message shows genuine evidence of being triggered by an
external bug/issue report, vs. simply carrying a PR/issue number because it
went through GitHub's standard PR workflow (which nearly every commit does
on a mature project) or a project-specific commit convention:

- 7/40 (17.5%) clear true positives — explicit "Fixes #N" / "Closes #N"
  language tied to a real reported problem
- 17/40 (42.5%) clear false positives — routine scheduled releases
  ("release: cut the v13.2.0-next.2 release (#44701)"), changelog bumps
  ("docs: changelog 5.13.1 (#46973)"), bot-authored contributor-list
  additions ("docs: add drakulavich as a contributor... [skip ci]"), and
  Node.js's universal `PR-URL:` commit trailer (applied to *every* commit
  in that project regardless of whether it was reactive) — none show
  genuine external triggering despite matching the `issue_number` or
  `github_pr_url` pattern
- 16/40 (40%) ambiguous — plausible either way from the message text alone

Strict precision (TP only): **17.5%**. Lenient precision (giving ambiguous
cases the benefit of the doubt): **57.5%**. This is a single-coder,
single-pass judgment call, not a substitute for proper double-blind
annotation — treat it as a warning sign that motivates fixing the heuristic
before resubmission, not as a final number to report. We also found at
least one clear **false negative** in a 30-commit proactive-sample check:
`spring-boot`'s "Closes gh-41098" uses GitHub's alternate short-reference
format (`gh-NNNN`), which the regex (`#\d+` only) misses entirely.

**What this means for the paper:** the *external artifact reference*
criterion (Section 5.1.2, Criterion 1) is too permissive — it fires on any
`#NNN`-shaped token regardless of context, which captures routine PR-merge
numbering (universal on GitHub) as readily as genuine issue-driven fixes.
Recommended before resubmission:
1. **Tighten Criterion 1** to require the issue/PR reference to co-occur
   with fix/bug-context language (extend the existing `fix_with_error_term`
   proximity-window approach to the bare `issue_number` pattern too, rather
   than treating a bare `#NNN` as sufficient on its own).
2. **Exclude routine/scheduled patterns** — release-cut commits
   (`^release:`, version-bump patterns), changelog-only commits, and
   bot-authored commits (join against the same bot list already used for
   the ownership pipeline) — from the reactive count entirely, since these
   are structurally certain to carry a PR number without being
   coordination-triggered.
3. **Run a proper validation pass**: 2 independent human coders label
   ~150-200 stratified-sampled commits, report Cohen's κ for
   inter-rater agreement, then report precision/recall of the automated
   heuristic against that ground truth — this is what reviewer B actually
   asked for, and a single-agent read-through (however careful) isn't a
   substitute.
4. **Reframe the construct**: even after tightening, consider renaming
   "reactive"/"external coordination linkage" to something more modest like
   "issue/PR-referenced," since "reactive" implies a causal triggering
   relationship the commit message alone can't fully establish.

This is a more consequential finding than the confounding-by-size issue in
Section 2 — it affects the *numerator* of the 49.0% statistic itself, not
just how the number is causally interpreted. Recommend treating this as a
priority fix before the SANER submission, ahead of the other, more cosmetic
items below.
